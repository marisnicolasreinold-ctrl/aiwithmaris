The 90-day test: how to set up an AI project that has to prove its value

11 June 2026 · 2 min read · Maris Reinold

“We need to give the project more time.” By the time someone says that sentence, it’s usually already too late. My counter-proposal: give the project less time — but a clear mandate. 90 days. Then the bill is settled.

Why 90 days, of all things?

Long enough to build something real and test it in everyday work. Short enough that nobody can hide behind “it’s still ramping up”. And short enough that failure doesn’t hurt — a buried 90-day project is a cheap lesson, a buried two-year project is a disaster.

Weeks 1–2: measure first, build second

The most common mistake: starting to build without knowing the status quo. If you don’t know how long an invoice takes today from arrival to booking, you won’t be able to say in 90 days whether things have improved.

So: measure the baseline first. How many cases per week? How much time per case? How many queries, how many errors? It doesn’t have to be scientific — a tally sheet over two weeks is often enough.

Define “passed” before you start

The second mistake: defining success only afterwards. Then every result gets talked up. Decide in advance how you’ll recognise the benefit — for example: “Throughput time is cut in half” or “The team saves noticeable hours per week, and the error rate doesn’t rise.”

The perfect metric isn’t the point. The point is that it’s on paper before you start.

Keep it small — genuinely small

A 90-day test can’t carry a full rollout. One process, one department, a handful of users who actually want it. Everything that doesn’t directly contribute to the proof gets cut: no integration with every system, no edge cases, no “while we’re at it”. The edge cases get their turn once the normal case has proven itself.

Day 90: one of three things happens

  1. The value is there — then you scale: more users, more cases, next process.
  2. The value is almost there — then there’s exactly one extension, with a concrete reason. One.
  3. The value isn’t there — then you bury it. No drama. Knowing what doesn’t work is honestly earned knowledge.

The beauty of this setup: it takes the religion out of the AI debate. It’s no longer about “do we believe in AI?” but about a question every craftsman understands instantly: has the tool proven itself — yes or no?
